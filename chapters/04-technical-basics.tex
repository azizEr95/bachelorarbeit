\chapter{Technische Grundlagen}

Aufbauend auf der im vorherigen Kapitel erarbeiteten Konzeption werden in diesem Kapitel die technischen Grundlagen erläutert, die für die Umsetzung von petappoint eingesetzt werden.

\section{Technologien im bereits bestehenden Backend}

Das Backend von petappoint wurde im Rahmen eines vorangegangenen Projektmoduls entwickelt und bildet die technische Grundlage für die in dieser Arbeit umgesetzte mobile Anwendung. Es basiert auf Node.js und Express, die gemeinsam eine REST-API bereitstellen, über die das Frontend sämtliche Daten abruft und verwaltet \cite{nodejs-docs, express-docs}. Als Datenbank kommt PostgreSQL zum Einsatz, das die ACID-Eigenschaften gewährleistet. ACID steht für Atomicity, Consistency, Isolation und Durability und beschreibt die Anforderungen an sichere und zuverlässige Datenbanktransaktionen \cite{postgresql_transactions}. Jegliche angefragte Änderungen werden entweder vollständig oder gar nicht ausgeführt, die Datenkonsistenz bleibt stets gewahrt, parallele Transaktionen beeinflussen sich nicht gegenseitig und abgeschlossene Transaktionen bleiben dauerhaft gespeichert \cite{postgresql_about}. Die Geschäftslogik ist durchgängig in TypeScript implementiert, wodurch Typsicherheit über alle Schichten der Anwendung sichergestellt wird \cite{typescript-docs}. Eingehende API-Anfragen werden mithilfe von Zod validiert \cite{zod-docs}. Für den automatisierten Versand von Buchungsbestätigungen und -erinnerungen wird Brevo als transaktionaler E-Mail-Dienst eingesetzt \cite{brevo-docs}.

\section{Plattformübergreifende Frameworks für mobile Anwendungen}

\subsection{React Native}

React Native ist ein von Meta entwickeltes Open-Source-Framework, das 
die Entwicklung mobiler Anwendungen für iOS und Android auf Basis von 
JavaScript bzw. TypeScript ermöglicht. Es rendert echte native UI-Komponenten, die direkt den nativen Komponenten der jeweiligen Plattform entsprechen und damit ein plattformkonformes Nutzererlebnis gewährleisten \cite{react-native-docs}. UI-Komponenten sind die grundlegenden Bausteine einer Benutzeroberfläche und bestehen aus mehreren UI-Elementen \cite{ui-elements} Dadurch verhalten sich die Elemente einer React Native Anwendung genauso wie die einer nativ entwickelten App mit iOS oder Android.

Ein Vorteil von React Native ist die Verwendung von JavaScript bzw. TypeScript, wodurch Entwickler mit bestehenden Webkenntnissen direkt in die mobile Entwicklung einsteigen können. Darüber hinaus ermöglicht eine einzige Codebasis die Entwicklung für iOS und Android gleichzeitig, was den Entwicklungsaufwand erheblich reduziert. Da React Native auf denselben Konzepten wie React aufbaut, können Kenntnisse und Codestrukturen zwischen Web und Mobile geteilt werden\cite{react-native-docs}.

Als Nachteil älterer Versionen ist zu nennen, dass React Native Anwendungen mit hohen aufwendigen Animationen oder Spielen an seine Grenzen stoßen konnte, da die Kommunikation zwischen 
JavaScript und der nativen Ebene einen gewissen Aufwand mit sich brachte. Mit der neuen Architektur in der das JavaScript Interface (JSI) hinzukam, wurde dieses Problem jedoch 
deutlich reduziert~\cite{react-native-new-arch}.

\subsection{Flutter}

Flutter ist ein von Google entwickeltes Open-Source-Framework für die plattformübergreifende Entwicklung mobiler Anwendungen. Es basiert auf 
der Programmiersprache Dart und verwendet eine eigene Rendering-Engine namens Impeller, die alle UI-Elemente auf einem Canvas (digitalen Zeichenfläche) selbst Pixel für Pixel zeichnet, anstatt die vom Betriebssystem bereitgestellten nativen Komponenten zu verwenden 
\cite{flutter-docs}. Ein Vorteil von Flutter ist die vollständige Kontrolle über das Erscheinungsbild der Anwendung, was für Produkte mit einem eigenen Design-System von Vorteil sein kann. Darüber hinaus unterstützt Flutter 
neben iOS und Android auch Web und Desktop aus einer einzigen Codebasis. Als Nachteil erfordert Flutter die Verwendung von Dart, die zwingend erforderlich ist und nicht durch JavaScript oder TypeScript ersetzt werden kann. Zudem entsprechen UI-Elemente nicht dem plattformspezifischen Design des Betriebssystems, da Flutter alles selbst zeichnet \cite{flutter-docs}.

\subsection{.NET MAUI}
.NET MAUI (Multi-platform App UI) ist ein von Microsoft entwickeltes Open-Source-Framework für die plattformübergreifende Entwicklung mobiler und Desktop-Anwendungen auf Basis von C\# und dem 
.NET-Ökosystem~\cite{maui-docs}. Es ist der offizielle Nachfolger von Xamarin, dessen Support im Mai 2024 eingestellt wurde~\cite{maui-docs}. Ein Vorteil von .NET MAUI ist die Nutzung nativer UI-Komponenten, sodass die Anwendung auf jeder Plattform dem vertrauten Design des 
Betriebssystems entspricht. Für Entwickler, die bereits Erfahrung mit C\# und dem .NET-Ökosystem haben, bietet .NET MAUI zudem den Vorteil, bestehende Kenntnisse direkt in der mobilen Entwicklung einsetzen zu können. Als Nachteil erfordert .NET MAUI einen vollständigen Sprachwechsel zu C\#, was für Teams ohne .NET-Erfahrung einen 
erheblichen Einarbeitungsaufwand bedeutet~\cite{maui-docs}.

\subsection{Auswertung und Begründung der Wahl von React Native}

React Native wurde als Framework für petappoint gewählt, da es die 
Anforderungen des Projekts in mehrfacher Hinsicht optimal erfüllt. 
Da das Backend bereits in TypeScript implementiert ist, ermöglicht 
React Native eine durchgängige Verwendung derselben Sprache über alle 
Schichten der Anwendung hinweg. Dadurch können Typdefinitionen und 
Validierungslogiken zwischen Backend und Frontend geteilt werden, was 
die Konsistenz der Codebasis erhöht und Fehlerquellen reduziert.

Ein weiterer entscheidender Faktor ist die Wiederverwendung bestehender 
React-Kenntnisse. Da React Native auf denselben Konzepten wie React 
aufbaut, entfällt ein vollständiger Technologiewechsel, was den 
Entwicklungsaufwand deutlich senkt. Im Vergleich zu Flutter, das die 
Verwendung von Dart voraussetzt, und Xamarin, das C\# und das 
.NET-Ökosystem erfordert, stellt React Native für JavaScript- und 
TypeScript-erfahrene Entwickler die niedrigste Einstiegshürde dar.

Da petappoint eine Terminbuchungs-App ist und keine aufwendigen 
Animationen oder grafikintensive Inhalte erfordert, spielen die 
performance-kritischen Grenzen von React Native in diesem Kontext 
keine entscheidende Rolle. Die nativen UI-Komponenten von React Native 
gewährleisten zudem ein plattformkonformes Nutzererlebnis auf iOS und 
Android, was für eine produktiv eingesetzte Anwendung von Bedeutung ist.

\section{Expo als Entwicklungsplattform und Routing-System für React Native}

Expo ist eine Open-Source-Plattform, die auf React Native aufbaut und 
die Entwicklung, das Testen und die Veröffentlichung mobiler Anwendungen 
vereinfacht. Sie stellt eine Sammlung von vorkonfigurierten Tools, 
Bibliotheken und Diensten bereit, die den Einstieg in die 
React-Native-Entwicklung erheblich erleichtern \cite{expo-docs}. Expo 
abstrahiert dabei viele der komplexen nativen Konfigurationsschritte 
und bietet stattdessen eine einheitliche Entwicklungsumgebung für 
iOS und Android .

Für das Testen auf einem physischen Gerät wurde in dieser Arbeit ein 
Development Build eingesetzt. Dabei wird ein individuell konfigurierter 
nativer Build der Anwendung erstellt, der den Expo Dev Client enthält 
und direkt auf dem Gerät installiert wird \cite{expo-build}. Dadruch wird die Verwendung beliebiger 
nativer Module erlaubt und spiegelt das Verhalten der finalen Produktionsanwendung 
deutlich genauer wider. Darüber hinaus bietet das Expo SDK eine 
umfangreiche Sammlung an vorgefertigten Modulen für den Zugriff auf 
native Gerätefunktionen wie Kamera, Standort oder Benachrichtigungen 
\cite{expo-build}.

Expo beinhaltet mit Expo Router ein dateibasiertes Navigations- und 
Routing-System, das speziell für React Native Anwendungen entwickelt 
wurde. Die Navigationsstruktur der Anwendung wird dabei direkt über 
das Dateiverzeichnis abgebildet, wobei jede Datei im \texttt{app}-Verzeichnis 
automatisch einer Route entspricht. Dieses Konzept ist an das Routing 
von Next.js angelehnt und ermöglicht eine intuitive und 
nachvollziehbare Projektstruktur \cite{expo-router}.

Für die Speicherung sicherheitsrelevanter Daten, wie beispielsweise 
Authentifizierungstokens, stellt das Expo SDK mit expo-secure-store 
eine integrierte Lösung bereit. Die Bibliothek legt Daten verschlüsselt 
als Key-Value-Paare ab und greift dabei direkt auf die systemeigenen 
Sicherheitsmechanismen der Endgeräte zurück, also die iOS-Keychain 
oder den Android-KeyStore \cite{expo-service}.

\subsection{Auswertung und Begründung der Wahl von Expo}

Expo wurde als Entwicklungsplattform für petappoint gewählt, da es 
nicht wirklich eine Alternative zu Expo gibt und der Entwicklungsprozess 
in mehrfacher Hinsicht vereinfacht wird. Durch die Abstraktion nativer 
Konfigurationsschritte kann der Fokus vollständig auf die Implementierung 
der Anwendungslogik gelegt werden. Diese Wahl wird zudem durch die 
offizielle React Native Dokumentation gestützt, die Expo ausdrücklich 
als empfohlenen Weg für die Entwicklung neuer React-Native-Anwendungen 
bezeichnet \cite{react-native-docs}. Da die Wahl bereits auf Expo 
gefallen war, ergab sich Expo Router als konsequente Folgeentscheidung, 
da es von Expo als offizielles Navigationssystem empfohlen wird 
\cite{expo-router} und die Nachvollziehbarkeit der Navigationslogik 
erhöht, indem die Ordnerstruktur direkt die Struktur der App widerspiegelt.

\section{Server-State-Management und Datenabruf}

\subsection{TanStack Query}

TanStack Query, früher bekannt als React Query, ist eine Bibliothek 
für das Management von Server-State in React- und React-Native-Anwendungen. 
Sie abstrahiert die Logik für das Abrufen, Caching, Synchronisieren und 
Aktualisieren von Serverdaten und ersetzt damit manuell verwaltete 
Ladezustände und Fehlerbehandlungen \cite{tanstack-query-docs}. 
Server-State bezeichnet dabei Daten, die auf einem externen Server 
gespeichert sind und über eine API abgerufen werden, im Gegensatz zum 
Client-State, der lokal in der Anwendung verwaltet wird.

Ein Vorteil von TanStack Query ist die automatische Verwaltung von 
Ladezuständen, Fehlerzuständen und dem Cache der abgerufenen Daten. 
Einmal abgerufene Daten werden zwischengespeichert und bei erneuten 
Anfragen direkt aus dem Cache zurückgegeben, bevor im Hintergrund eine 
Aktualisierung stattfindet. Dieses Verhalten wird als 
Stale-While-Revalidate bezeichnet und sorgt für eine reaktionsschnelle 
Benutzeroberfläche \cite{tanstack-query-docs}. Darüber hinaus reduziert 
TanStack Query den manuellen Implementierungsaufwand erheblich, da 
wiederkehrende Muster wie Fehlerbehandlung und Ladezustände nicht 
selbst implementiert werden müssen.

Als Nachteil ist zu nennen, dass TanStack Query ausschließlich für 
Server-State konzipiert ist und keine Lösung für die Verwaltung von 
lokalem Client-State bietet. Für Anwendungen, die sowohl umfangreichen 
Server-State als auch komplexen Client-State verwalten müssen, ist 
daher eine zusätzliche State-Management-Lösung erforderlich \cite{tanstack-query-docs}.

\subsection{Redux Toolkit}

Redux Toolkit ist die offizielle, empfohlene Variante zur Verwendung 
von Redux, einer weit verbreiteten Bibliothek für das globale 
State-Management in React-Anwendungen \cite{redux-toolkit-docs}. 
Redux verwaltet den gesamten Anwendungszustand in einem zentralen Store, auf den alle Komponenten zugreifen können.

Ein Vorteil von Redux Toolkit ist die große Community und die weite 
Verbreitung in bestehenden Projekten. Zudem bietet Redux Toolkit mit ihrem "RTK Query" eine integrierte Lösung für den Datenabruf und das Caching 
von Serverdaten, wodurch sowohl Client-State als auch Server-State 
in einem einzigen Tool verwaltet werden können.

Als Nachteil gilt der vergleichsweise hohe Boilerplate-Aufwand, auch 
in der Toolkit-Variante. Für kleinere Projekte kann der konfigurationsintensive 
Ansatz von Redux im Verhältnis zum tatsächlichen Nutzen unverhältnismäßig 
aufwendig sein.

\subsection{Zustand}

Zustand ist eine minimalistische State-Management-Bibliothek für React, 
die ohne viel Boilerplate auskommt und einen einfachen, flexiblen Ansatz für die Verwaltung von globalem Client-State bietet \cite{zustand-docs}.

Ein Vorteil von Zustand ist die geringe Einstiegshürde und die schlanke 
API, die eine schnelle Integration in bestehende Projekte ermöglicht. 
Im Vergleich zu Redux Toolkit erfordert Zustand deutlich weniger 
Konfigurationsaufwand und eignet sich daher besonders für kleinere 
bis mittelgroße Anwendungen.

Als Nachteil ist zu nennen, dass Zustand wie Redux primär für 
Client-State ausgelegt ist und keine eingebauten Mechanismen für 
Caching oder Synchronisierung von Serverdaten bietet. Für den 
Datenabruf ist daher stets eine zusätzliche Lösung erforderlich.

\subsection{Auswertung und Begründung der Wahl von TanStack Query}

Die Wahl fiel auf TanStack Query, da die Bibliothek bereits im 
vorangegangenen Projektmodul eingesetzt wurde und somit Vorerfahrung 
im Umgang damit vorhanden war. Darüber hinaus wurde TanStack Query 
vom betreuenden Professor empfohlen. Technisch überzeugt die 
Bibliothek durch ihr spezialisiertes Server-State-Management und die 
deutliche Reduzierung von Boilerplate-Code gegenüber einer manuellen 
Implementierung. Da petappoint eine datengetriebene Anwendung ist, 
die primär Serverdaten abruft und anzeigt, ist eine spezialisierte 
Server-State-Lösung einer allgemeinen State-Management-Bibliothek 
vorzuziehen \cite{tanstack-query-docs}.

Redux Toolkit schied aus, da es einen höheren Konfigurationsaufwand 
mit sich bringt und Redux nicht bereits im Projekt eingesetzt wurde, 
wodurch kein Mehrwert gegenüber TanStack Query entstanden wäre. 
Zustand wurde nicht gewählt, da es primär für lokalen Client-State 
ausgelegt ist und keine spezialisierte Unterstützung für Server-State 
bietet.

\section{UI-Komponentenbibliotheken}

\subsection{Gluestack UI}

Gluestack UI ist eine komponentenbasierte UI-Bibliothek für React 
Native, die eine Sammlung von vordefinierten, zugänglichen und 
anpassbaren Komponenten bereitstellt. Die Bibliothek basiert auf 
einem Token-basierten Design-System, bei dem Farben, Abstände und 
Typografie über zentrale Design-Tokens definiert und konsistent in 
der gesamten Anwendung verwendet werden \cite{gluestack-docs}. 
Gluestack UI unterstützt sowohl React Native als auch React im Web, 
wodurch Komponenten plattformübergreifend eingesetzt werden können.

Ein Vorteil von Gluestack UI ist die umfangreiche Sammlung an 
vorgefertigten Komponenten, die direkt eingesetzt und an das 
eigene Design angepasst werden können. Das Token-basierte 
Design-System ermöglicht eine konsistente visuelle Gestaltung 
der gesamten Anwendung ohne redundante Styling-Definitionen. 
Darüber hinaus sind alle Komponenten auf Barrierefreiheit 
ausgelegt und folgen den gängigen Accessibility-Standards.

Als Nachteil ist zu nennen, dass Gluestack UI eine verhältnismäßig 
junge Bibliothek ist und sich die API zwischen Versionen teils 
erheblich verändert hat, was zu Migrationsaufwand führen kann. 
Zudem ist die Community im Vergleich zu etablierteren Bibliotheken 
wie React Native Paper noch kleiner, weshalb die Dokumentation 
in manchen Bereichen weniger ausführlich ist.

\subsection{React Native Paper}

React Native Paper ist eine weit verbreitete UI-Komponentenbibliothek 
für React Native, die auf den Material Design Richtlinien von Google 
basiert \cite{react-native-paper-docs}. Sie stellt eine Vielzahl 
von vorgefertigten Komponenten bereit, die dem Material Design 
Erscheinungsbild entsprechen und auf beiden Plattformen konsistent 
aussehen.

Ein Vorteil von React Native Paper ist die große Community und die 
ausgereifte, stabile API, die sich über mehrere Jahre bewährt hat. 
Die Bibliothek ist gut dokumentiert und bietet eine zuverlässige 
Grundlage für Anwendungen, die dem Material Design folgen sollen.

Als Nachteil ist zu nennen, dass React Native Paper stark an das 
Material Design gebunden ist, was die gestalterische Freiheit 
einschränkt. Anwendungen, die ein eigenes Design-System oder ein 
plattformkonformes iOS-Design anstreben, stoßen mit React Native 
Paper schnell an ihre Grenzen.

\subsection{StyleSheet}

StyleSheet ist die in React Native integrierte Lösung für das Styling 
von Komponenten \cite{react-native-docs}. Es handelt sich dabei um 
keine externe Bibliothek, sondern um eine native API, die CSS-ähnliche 
Styling-Eigenschaften in JavaScript-Objekte übersetzt und diese 
optimiert an die native Rendering-Ebene weitergibt.

Ein Vorteil von StyleSheet ist, dass keine zusätzliche Abhängigkeit 
eingebunden werden muss, da es direkt in React Native enthalten ist. 
Die Performance ist durch die native Optimierung der Styles 
vergleichsweise hoch.

Als Nachteil ist zu nennen, dass StyleSheet keine vorgefertigten 
Komponenten bereitstellt. Sämtliche UI-Elemente müssen von Grund 
auf selbst implementiert werden, was den Entwicklungsaufwand 
erheblich erhöht. Zudem fehlen ein einheitliches Design-System 
und Mechanismen für die konsistente Verwaltung von Design-Tokens.

\subsection{Auswertung und Begründung der Wahl von Gluestack UI}

Gluestack UI wurde als UI-Komponentenbibliothek für petappoint gewählt, 
da es eine umfangreiche Sammlung anpassbarer Komponenten mit einem 
konsistenten Token-basierten Design-System vereint. Im Vergleich zu 
StyleSheet entfällt der Aufwand, alle UI-Elemente von Grund auf selbst 
zu implementieren, was die Entwicklungsgeschwindigkeit erheblich 
steigert. Gegenüber React Native Paper bietet Gluestack UI mehr 
gestalterische Freiheit, da es nicht an ein spezifisches Design-System 
wie Material Design gebunden ist und somit ein individuelles 
Erscheinungsbild der Anwendung ermöglicht \cite{gluestack-docs}.

\section{TypeScript und Zod}
Da TypeScript bereits im Backend eingesetzt wird, entstand durch die Verwendung im Frontend eine konsistente Typisierung über alle Schichten der Anwendung hinweg. Dasselbe gilt für Zod, da es ebenfalls bereits im Backend eingesetzt wird, können Schemata zwischen Backend und Frontend geteilt werden, was die Konsistenz der Datenvalidierung über die gesamte Anwendung hinweg gewährleistet.

\section{Internationalisierung mit i18next}

i18next ist eine Internationalisierungsbibliothek für JavaScript, die 
in Kombination mit react-i18next für React Native Anwendungen eingesetzt 
wird. Sie ermöglicht die Verwaltung von Übersetzungsdateien und passt 
die Sprache der Oberfläche automatisch an die Systemsprache des Geräts 
an \cite{i18n-docs}.

Im Vergleich zu react-intl, einer weiteren verbreiteten 
Internationalisierungsbibliothek, zeichnet sich i18next durch eine 
flexible Konfiguration und eine größere Community aus. react-intl ist 
eng an den ECMAScript Internationalization Standard gebunden und eignet 
sich besonders für Web-Anwendungen, bietet jedoch weniger Flexibilität 
bei der Integration in React Native \cite{i18n-docs}.

Die Wahl fiel auf i18next, da es sich als Standardlösung im React 
Native Ökosystem etabliert hat und die automatische Anpassung an die 
Systemsprache des Geräts ohne zusätzlichen Konfigurationsaufwand 
ermöglicht.

\section{Gesamtauswertung der Technologieauswahl}

Die Wahl der eingesetzten Technologien orientierte sich sowohl an 
technischen als auch an projektspezifischen Anforderungen. Mit React 
Native und Expo wurde eine Kombination gewählt, die von der offiziellen 
React Native Dokumentation ausdrücklich empfohlen wird und die 
plattformübergreifende Entwicklung für iOS und Android aus einer 
einzigen Codebasis ermöglicht. Da das Backend bereits in TypeScript 
implementiert ist, konnte durch die Wahl von React Native eine 
durchgängige Verwendung derselben Sprache über alle Schichten der 
Anwendung hinweg sichergestellt werden.

TanStack Query wurde aufgrund von Vorerfahrung aus dem vorangegangenen 
Projektmodul sowie der Empfehlung des betreuenden Professors gewählt. 
Technisch überzeugt die Bibliothek durch ihr spezialisiertes 
Server-State-Management, das den Anforderungen einer datengetriebenen 
Terminbuchungs-App wie petappoint optimal entspricht. Gluestack UI 
ermöglicht eine konsistente und anpassbare Gestaltung der 
Benutzeroberfläche ohne Bindung an ein spezifisches Design-System 
wie Material Design. TypeScript und Zod werden durchgängig im Frontend 
eingesetzt und ermöglichen die gemeinsame Nutzung von Typdefinitionen 
und Validierungsschemata mit dem Backend. i18next stellt die 
Mehrsprachigkeit der Anwendung sicher und passt die Sprache der 
Oberfläche automatisch an die Systemsprache des Geräts an.

Insgesamt wurde bei der Technologieauswahl darauf geachtet, bewährte 
und gut dokumentierte Lösungen einzusetzen, die sich im React Native 
Ökosystem etabliert haben und den Entwicklungsaufwand durch hohe 
Abstraktion und vorgefertigte Funktionalitäten minimieren.
