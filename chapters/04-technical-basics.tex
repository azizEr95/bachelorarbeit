\chapter{Technische Grundlagen}

Aufbauend auf der im vorherigen Kapitel erarbeiteten Konzeption werden in diesem Kapitel die technischen Grundlagen erläutert, die für die Umsetzung von \textit{petappoint} eingesetzt werden.

\section{Technologien im bereits bestehenden Backend}

Das Backend von \textit{petappoint} wurde im Rahmen eines vorangegangenen Projektmoduls entwickelt und bildet die technische Grundlage für die in dieser Arbeit umgesetzte mobile Anwendung. Es basiert auf Node.js und Express, die gemeinsam eine REST-API bereitstellen, über die das Frontend sämtliche Daten abruft und verwaltet \cite{nodejs-docs, express-docs}. Als Datenbank kommt PostgreSQL zum Einsatz, das die ACID-Eigenschaften gewährleistet. ACID steht für Atomicity, Consistency, Isolation und Durability und beschreibt die Anforderungen an sichere und zuverlässige Datenbanktransaktionen \cite{postgresql_transactions}. Jegliche angefragte Änderungen werden entweder vollständig oder gar nicht ausgeführt, die Datenkonsistenz bleibt stets gewahrt, parallele Transaktionen beeinflussen sich nicht gegenseitig und abgeschlossene Transaktionen bleiben dauerhaft gespeichert \cite{postgresql_about}. Die Geschäftslogik ist durchgängig in TypeScript implementiert, wodurch Typsicherheit über alle Schichten der Anwendung sichergestellt wird \cite{typescript-docs}. Eingehende API-Anfragen werden mithilfe von Zod validiert \cite{zod-docs}. Für den automatisierten Versand von Buchungsbestätigungen und -erinnerungen wird Brevo als transaktionaler E-Mail-Dienst eingesetzt \cite{brevo-docs}.

\section{Plattformübergreifende Frameworks für mobile Anwendungen}

\subsection{React Native}

React Native ist ein von Meta entwickeltes Open-Source-Framework, das die Entwicklung mobiler Anwendungen für iOS und Android auf Basis von JavaScript bzw. TypeScript ermöglicht. Es rendert echte native UI-Komponenten, die direkt den nativen Komponenten der jeweiligen Plattform entsprechen und damit ein plattformkonformes Nutzererlebnis gewährleisten \cite{react-native-docs}. UI-Komponenten sind die grundlegenden Bausteine einer Benutzeroberfläche und bestehen aus mehreren UI-Elementen \cite[S.~4]{galitz2002}. Dadurch verhalten sich die Elemente einer React Native Anwendung genauso wie die einer nativ entwickelten Anwendung mit iOS oder Android. Da das Backend bereits in TypeScript implementiert ist, ermöglicht React Native zudem eine durchgängige Verwendung derselben Sprache über alle Schichten der Anwendung hinweg, wodurch Typdefinitionen zwischen Backend und Frontend geteilt werden können. Darüber hinaus ermöglicht eine einzige Codebasis die Entwicklung für iOS und Android gleichzeitig, was den Entwicklungsaufwand erheblich reduziert. Da React Native auf denselben Konzepten wie React aufbaut, können Kenntnisse und Codestrukturen zwischen Web und Mobile geteilt werden \cite{react-native-docs}.

Als Nachteil älterer Versionen ist zu nennen, dass React Native Anwendungen mit hohen aufwendigen Animationen oder Spielen an seine Grenzen stoßen konnte, da die Kommunikation zwischen 
JavaScript und der nativen Ebene einen gewissen Aufwand mit sich brachte. Mit der neuen Architektur in der das JavaScript Interface (JSI) hinzukam, wurde dieses Problem jedoch 
deutlich reduziert~\cite{react-native-new-arch}.

\subsection{Flutter}

Flutter ist ein von Google entwickeltes Open-Source-Framework für die plattformübergreifende Entwicklung mobiler Anwendungen. Es basiert auf der Programmiersprache Dart und verwendet eine eigene Rendering-Engine namens Impeller, die alle UI-Elemente auf einem Canvas (digitalen Zeichenfläche) selbst Pixel für Pixel zeichnet, anstatt die vom Betriebssystem bereitgestellten nativen Komponenten zu verwenden \cite{flutter-docs}. Ein weiterer Vorteil von Flutter ist die vollständige Kontrolle über das Erscheinungsbild der Anwendung, was für Produkte mit einem eigenen Design-System von Vorteil sein kann. Darüber hinaus unterstützt Flutter neben iOS und Android auch Web und Desktop aus einer einzigen Codebasis. 

Als Nachteil erfordert Flutter die Verwendung von Dart, die zwingend erforderlich ist und nicht durch JavaScript oder TypeScript ersetzt werden kann. Zudem entsprechen UI-Elemente nicht dem plattformspezifischen Design des Betriebssystems, da Flutter alles selbst zeichnet \cite{flutter-docs}.

\subsection{.NET MAUI}
.NET MAUI (Multi-platform App UI) ist ein von Microsoft entwickeltes Open-Source-Framework für die plattformübergreifende Entwicklung mobiler und Desktop-Anwendungen auf Basis von C\# und dem 
.NET-Ökosystem~\cite{maui-docs}. Es ist der offizielle Nachfolger von Xamarin, dessen Support im Mai 2024 eingestellt wurde~\cite{maui-docs}. Ein Vorteil von .NET MAUI ist die Nutzung nativer UI-Komponenten, sodass die Anwendung auf jeder Plattform dem vertrauten Design des 
Betriebssystems entspricht. Für Entwickler, die bereits Erfahrung mit C\# und dem .NET-Ökosystem haben, bietet .NET MAUI zudem den Vorteil, bestehende Kenntnisse direkt in der mobilen Entwicklung einsetzen zu können. 

Als Nachteil erfordert .NET MAUI einen vollständigen Sprachwechsel zu C\#, was für Teams ohne .NET-Erfahrung einen 
erheblichen Einarbeitungsaufwand bedeutet~\cite{maui-docs}.

\subsection{Begründung der Wahl von React Native}

React Native wurde als Framework für \textit{petappoint} gewählt, da es durch die gemeinsame Verwendung von TypeScript mit dem Backend eine konsistente Typisierung über alle Schichten der Anwendung ermöglicht. Dadurch können Typdefinitionen und 
Validierungslogiken zwischen Backend und Frontend geteilt werden, was die Konsistenz der Codebasis erhöht und Fehlerquellen reduziert. Zudem entfällt durch die Nähe zu React ein vollständiger Technologiewechsel, was den Entwicklungsaufwand gegenüber Flutter und .NET MAUI deutlich senkt. Da \textit{petappoint} keine aufwendigen Animationen erfordert, spielen die früheren performance-kritischen Grenzen von React Native keine entscheidende Rolle.

\section{Expo als empfohlene Entwicklungsplattform für React Native}

Expo ist eine Open-Source-Plattform, die auf React Native aufbaut und die Entwicklung, das Testen und die Veröffentlichung mobiler Anwendungen vereinfacht. Sie stellt eine Sammlung von vorkonfigurierten Tools, Bibliotheken und Diensten bereit, die den Einstieg in die React Native Entwicklung erheblich erleichtern \cite{expo-docs}. Expo abstrahiert dabei viele der komplexen nativen Konfigurationsschritte und bietet stattdessen eine einheitliche Entwicklungsumgebung für iOS und Android.

Expo wurde als Entwicklungsplattform für \textit{petappoint} gewählt, da die offizielle React Native Dokumentation Expo ausdrücklich als empfohlenen Weg für die Entwicklung neuer React-Native-Anwendungen bezeichnet \cite{react-native-docs}. 

Da die Wahl dadurch bereits gefallen war, ergab sich Expo Router als konsequente Folgeentscheidung, 
da es von Expo als offizielles Navigationssystem empfohlen wird \cite{expo-router}. Die Navigationsstruktur der Anwendung wird dabei direkt über das Dateiverzeichnis abgebildet, wobei jede Datei im \texttt{app}-Verzeichnis automatisch einer Route entspricht. Dadurch wird Nachvollziehbarkeit der Navigationslogik erhöht, indem die Ordnerstruktur direkt die Struktur der App widerspiegelt. 

Um weiterhin in der Entwicklungsumgebung von Expo zu bleiben wurde die Anwendung für das Testen auf einem physischen Gerät durch ein Development Build eingesetzt. Dabei wird ein individuell konfigurierter nativer Build der Anwendung erstellt, der den Expo Dev Client enthält und direkt auf dem Gerät installiert wird \cite{expo-build}. Dadurch wird die Verwendung beliebiger nativer Module erlaubt und spiegelt das Verhalten der finalen Produktionsanwendung deutlich genauer wider. Das Expo Software Development Kit (Expo SDK) bietet darüber hinaus eine Sammlung an vorgefertigten Modulen für den Zugriff auf native Gerätefunktionen wie Kamera, Standort oder Benachrichtigungen \cite{expo-build}.


Für die Speicherung sicherheitsrelevanter Daten, wie beispielsweise Authentifizierungstokens, stellt das Expo SDK mit expo-secure-store eine integrierte Lösung bereit. Die Bibliothek legt Daten verschlüsselt als Key-Value-Paare ab und greift dabei direkt auf die systemeigenen Sicherheitsmechanismen der Endgeräte zurück, also die iOS-Keychain oder den Android-KeyStore \cite{expo-service}.

\section{Server-State-Management und Datenabruf}

\subsection{TanStack Query}

TanStack Query, früher bekannt als React Query, ist eine Bibliothek 
für das Management von Server-State in React- und React-Native-Anwendungen. 
Sie abstrahiert die Logik für das Abrufen, Caching, Synchronisieren und 
Aktualisieren von Serverdaten und ersetzt damit manuell verwaltete 
Ladezustände und Fehlerbehandlungen \cite{tanstack-query-docs}. 
Server-State bezeichnet dabei Daten, die auf einem externen Server 
gespeichert sind und über eine API abgerufen werden, im Gegensatz zum 
Client-State, der lokal in der Anwendung verwaltet wird.

Ein Vorteil von TanStack Query ist die automatische Verwaltung von 
Ladezuständen, Fehlerzuständen und dem Cache der abgerufenen Daten. 
Einmal abgerufene Daten werden zwischengespeichert und bei erneuten 
Anfragen direkt aus dem Cache zurückgegeben, bevor im Hintergrund eine 
Aktualisierung stattfindet. Dieses Verhalten wird als 
Stale-While-Revalidate bezeichnet und sorgt für eine reaktionsschnelle 
Benutzeroberfläche \cite{tanstack-query-docs}. Darüber hinaus reduziert 
TanStack Query den manuellen Implementierungsaufwand erheblich, da 
wiederkehrende Muster wie Fehlerbehandlung und Ladezustände nicht 
selbst implementiert werden müssen.

Als Nachteil ist zu nennen, dass TanStack Query ausschließlich für 
Server-State konzipiert ist und keine Lösung für die Verwaltung von 
lokalem Client-State bietet. Für Anwendungen, die sowohl umfangreichen 
Server-State als auch komplexen Client-State verwalten müssen, ist 
daher eine zusätzliche State-Management-Lösung erforderlich \cite{tanstack-query-docs}.

\subsection{SWR (Stale-While-Revalidate)}

Stale-While-Revalidate (SWR) ist eine von Vercel entwickelte Bibliothek für das Server-State-Management in React- und React-Native-Anwendungen. Der Name leitet sich von
Stale-While-Revalidate-Prinzip ab, das auch SWR zugrunde liegt. Das Prinzip funktioniert so, dass gecachte Daten sofort zurückgegeben werden, während im Hintergrund eine Aktualisierung stattfindet \cite{swr-docs}.

Ein Vorteil von SWR ist die schlanke API und die geringe Paketgröße, was eine schnelle Integration in bestehende Projekte ermöglicht. Für einfachere Anwendungen mit überschaubaren Datenanforderungen bietet SWR eine leichtgewichtige Alternative zu TanStack Query.

Als Nachteil bietet SWR einen deutlich kleineren Funktionsumfang als TanStack Query. Fortgeschrittene Features wie optimistische Updates, unendliches Scrollen oder die detaillierte Konfiguration von Cache-Strategien sind bei SWR weniger ausgereift oder nicht vorhanden 
\cite{swr-docs}.

\subsection{Redux Toolkit Query}

Redux Toolkit Query (RTK Query) ist eine integrierte Lösung für den Datenabruf und das Caching von Serverdaten \cite{redux-toolkit-docs}. Sie kombiniert Server-State-Management und Client-State-Management in einem einzigen Tool und ist damit eng mit dem globalen Redux Store 
verknüpft.

Ein Vorteil von RTK Query ist die enge Integration mit Redux, wodurch Server- und Client-State zentral in einem Store verwaltet werden können. Für Projekte, die bereits Redux einsetzen, entfällt durch RTK Query 
eine zusätzliche Abhängigkeit.

Als Nachteil setzt RTK Query die Verwendung von Redux voraus, was einen erheblichen Konfigurationsaufwand mit sich bringt. Für Projekte ohne bestehende Redux-Integration ist dieser Aufwand im Verhältnis zum Nutzen unverhältnismäßig hoch \cite{redux-toolkit-docs}.

\subsection{Begründung der Wahl von TanStack Query}

TanStack Query wurde als Server-State-Lösung für \textit{petappoint} gewählt, da es durch sein spezialisiertes Server-State-Management und die automatische Verwaltung von Ladezuständen und Caching den Implementierungsaufwand erheblich reduziert. Gegenüber SWR bietet TanStack Query einen deutlich größeren Funktionsumfang, und gegenüber RTK Query entfällt der hohe Konfigurationsaufwand einer vollständigen Redux-Integration. Darüber hinaus war TanStack Query bereits im vorangegangenen Projektmodul im Einsatz, wodurch Vorerfahrung im Umgang mit der Bibliothek vorhanden war.

\section{Client-State-Management}

\subsection{Zustand}

Zustand ist eine minimalistische State-Management-Bibliothek für React, die ohne viel Boilerplate auskommt und einen einfachen, flexiblen Ansatz für die Verwaltung von globalem Client-State bietet \cite{zustand-docs}.

Ein Vorteil von Zustand ist die geringe Einstiegshürde und die schlanke API, die eine schnelle Integration in bestehende Projekte ermöglicht. Im Vergleich zu Redux Toolkit erfordert Zustand deutlich weniger Konfigurationsaufwand und eignet sich daher besonders für kleinere bis mittelgroße Anwendungen.

Als Nachteil ist zu nennen, dass Zustand keine eingebauten Mechanismen für Caching oder Synchronisierung von Serverdaten bietet. Für den Datenabruf ist daher stets eine zusätzliche Lösung erforderlich 
\cite{zustand-docs}.

\subsection{Redux Toolkit}

Redux Toolkit ist die offizielle, empfohlene Variante zur Verwendung von Redux, einer weit verbreiteten Bibliothek für das globale State-Management in React-Anwendungen \cite{redux-toolkit-docs}. Redux verwaltet den gesamten Anwendungszustand in einem zentralen Store, auf den alle Komponenten zugreifen können.

Ein Vorteil von Redux Toolkit ist die große Community und die weite Verbreitung in bestehenden Projekten. Für sehr große Anwendungen mit komplexem globalem State bietet Redux Toolkit durch seinen strukturierten Ansatz eine robuste und gut skalierbare Lösung.

Als Nachteil gilt der vergleichsweise hohe Boilerplate-Aufwand, auch in der Toolkit-Variante. Für kleinere Projekte kann der konfigurationsintensive Ansatz von Redux im Verhältnis zum tatsächlichen Nutzen unverhältnismäßig aufwendig sein \cite{redux-toolkit-docs}.

\subsection{Begründung der Wahl von Zustand}

Zustand wurde als Client-State-Lösung für \textit{petappoint} gewählt, da es durch seine schlanke API und den geringen Konfigurationsaufwand optimal zur Größe und Komplexität der Anwendung passt. Gegenüber Redux Toolkit entfällt der hohe Boilerplate-Aufwand, was für eine mittelgroße Anwendung wie \textit{petappoint} verhältnismäßiger ist. Da \textit{petappoint} primär Serverdaten verwaltet und nur wenig lokalen Client-State benötigt, ist eine schlanke Lösung wie Zustand einer umfangreichen Bibliothek 
wie Redux Toolkit vorzuziehen.

\section{UI-Komponentenbibliotheken}

\subsection{Gluestack UI}

Gluestack UI ist eine komponentenbasierte UI-Bibliothek für React 
Native, die eine Sammlung von vordefinierten, zugänglichen und 
anpassbaren Komponenten bereitstellt. Die Bibliothek basiert auf 
einem Token-basierten Design-System, bei dem Farben, Abstände und 
Typografie über zentrale Design-Tokens definiert und konsistent in 
der gesamten Anwendung verwendet werden \cite{gluestack-docs}. 
Gluestack UI unterstützt sowohl React Native als auch React im Web, 
wodurch Komponenten plattformübergreifend eingesetzt werden können.

Ein Vorteil von Gluestack UI ist die umfangreiche Sammlung an 
vorgefertigten Komponenten, die direkt eingesetzt und an das 
eigene Design angepasst werden können. Das Token-basierte 
Design-System ermöglicht eine konsistente visuelle Gestaltung 
der gesamten Anwendung ohne redundante Styling-Definitionen. 
Darüber hinaus sind alle Komponenten auf Barrierefreiheit 
ausgelegt und folgen den gängigen Accessibility-Standards.

Als Nachteil ist zu nennen, dass Gluestack UI eine verhältnismäßig 
junge Bibliothek ist und sich die API zwischen Versionen teils 
erheblich verändert hat, was zu Migrationsaufwand führen kann. 
Zudem ist die Community im Vergleich zu etablierteren Bibliotheken 
wie React Native Paper noch kleiner, weshalb die Dokumentation 
in manchen Bereichen weniger ausführlich ist.

\subsection{React Native Paper}

React Native Paper ist eine weit verbreitete UI-Komponentenbibliothek 
für React Native, die auf den Material Design Richtlinien von Google 
basiert \cite{react-native-paper-docs}. Sie stellt eine Vielzahl 
von vorgefertigten Komponenten bereit, die dem Material Design 
Erscheinungsbild entsprechen und auf beiden Plattformen konsistent 
aussehen.

Ein Vorteil von React Native Paper ist die große Community und die 
ausgereifte, stabile API, die sich über mehrere Jahre bewährt hat. 
Die Bibliothek ist gut dokumentiert und bietet eine zuverlässige 
Grundlage für Anwendungen, die dem Material Design folgen sollen.

Als Nachteil ist zu nennen, dass React Native Paper stark an das 
Material Design gebunden ist, was die gestalterische Freiheit 
einschränkt. Anwendungen, die ein eigenes Design-System oder ein 
plattformkonformes iOS-Design anstreben, stoßen mit React Native 
Paper schnell an ihre Grenzen.

\subsection{StyleSheet}

StyleSheet ist die in React Native integrierte Lösung für das Styling 
von Komponenten \cite{react-native-docs}. Es handelt sich dabei um 
keine externe Bibliothek, sondern um eine native API, die CSS-ähnliche 
Styling-Eigenschaften in JavaScript-Objekte übersetzt und diese 
optimiert an die native Rendering-Ebene weitergibt.

Ein Vorteil von StyleSheet ist, dass keine zusätzliche Abhängigkeit 
eingebunden werden muss, da es direkt in React Native enthalten ist. 
Die Performance ist durch die native Optimierung der Styles 
vergleichsweise hoch.

Als Nachteil ist zu nennen, dass StyleSheet keine vorgefertigten 
Komponenten bereitstellt. Sämtliche UI-Elemente müssen von Grund 
auf selbst implementiert werden, was den Entwicklungsaufwand 
erheblich erhöht. Zudem fehlen ein einheitliches Design-System 
und Mechanismen für die konsistente Verwaltung von Design-Tokens.

\subsection{Begründung der Wahl von Gluestack UI}

Gluestack UI wurde als UI-Komponentenbibliothek für \textit{petappoint} gewählt, 
da es eine umfangreiche Sammlung anpassbarer Komponenten mit einem 
konsistenten Token-basierten Design-System vereint. Im Vergleich zu 
StyleSheet entfällt der Aufwand, alle UI-Elemente von Grund auf selbst 
zu implementieren, was die Entwicklungsgeschwindigkeit erheblich 
steigert. Gegenüber React Native Paper bietet Gluestack UI mehr 
gestalterische Freiheit, da es nicht an ein spezifisches Design-System 
wie Material Design gebunden ist und somit ein individuelles 
Erscheinungsbild der Anwendung ermöglicht \cite{gluestack-docs}.

\section{Internationalisierung}

\subsection{i18next}

i18next ist eine Internationalisierungsbibliothek für JavaScript, die 
in Kombination mit react-i18next für React Native Anwendungen eingesetzt 
wird. Sie ermöglicht die Verwaltung von Übersetzungsdateien und passt 
die Sprache der Oberfläche automatisch an die Systemsprache des Geräts 
an \cite{i18n-docs}.

Ein Vorteil von i18next ist die flexible Konfiguration und die große 
Community, die sich als Standardlösung im React Native Ökosystem 
etabliert hat. Die automatische Anpassung an die Systemsprache des 
Geräts erfolgt ohne zusätzlichen Konfigurationsaufwand, was die 
Integration erheblich vereinfacht.

Als Nachteil ist zu nennen, dass i18next durch seinen großen 
Funktionsumfang für einfache Anwendungsfälle überdimensioniert 
wirken kann. Die initiale Konfiguration erfordert zudem etwas 
mehr Einarbeitungszeit als bei schlankeren Alternativen \cite{i18n-docs}.

\subsection{react-intl}

react-intl ist eine Internationalisierungsbibliothek, die eng an den 
ECMAScript Internationalization Standard gebunden ist und von der 
FormatJS-Bibliotheksfamilie bereitgestellt wird \cite{react-intl-docs}. 
Sie bietet eine standardkonforme Lösung für die Formatierung von 
Datum, Uhrzeit, Zahlen und Übersetzungen.

Ein Vorteil von react-intl ist die enge Anlehnung an den ECMAScript 
Standard, was eine hohe Konsistenz bei der Formatierung von Datum, 
Uhrzeit und Zahlen gewährleistet. Für Web-Anwendungen, die eine 
standardkonforme Internationalisierung benötigen, ist react-intl 
eine ausgereifte und weit verbreitete Lösung.

Als Nachteil bietet react-intl weniger Flexibilität bei der Integration 
in React Native, da die Bibliothek primär für Web-Anwendungen ausgelegt 
ist. Die automatische Anpassung an die Systemsprache des Geräts erfordert 
zudem zusätzlichen Konfigurationsaufwand \cite{react-intl-docs}.

\subsection{Begründung der Wahl von i18next}

i18next wurde als Internationalisierungslösung für \textit{petappoint} gewählt, 
da es sich als Standardlösung im React Native Ökosystem etabliert hat 
und die automatische Anpassung an die Systemsprache des Geräts ohne 
zusätzlichen Konfigurationsaufwand ermöglicht. Gegenüber react-intl 
bietet i18next eine bessere Integration in React Native und mehr 
Flexibilität bei der Konfiguration.

\section{Zusammenspiel der gewählten Technologien}

Die Wahl der eingesetzten Technologien orientierte sich sowohl an 
technischen als auch an projektspezifischen Anforderungen. Ein 
zentrales Merkmal des gewählten Stacks ist die durchgängige Verwendung 
von TypeScript über alle Schichten der Anwendung hinweg. Da TypeScript 
bereits im Backend eingesetzt wird, können Typdefinitionen und 
Validierungsschemata mit Zod zwischen Backend und Frontend geteilt 
werden, was die Konsistenz der Codebasis erhöht und Fehlerquellen 
reduziert.

React Native und Expo bilden die Grundlage der Anwendung und ergänzen 
sich dabei gegenseitig: React Native ermöglicht die plattformübergreifende 
Entwicklung für iOS und Android aus einer einzigen TypeScript-Codebasis, 
während Expo die native Konfiguration abstrahiert und den Fokus auf 
die Anwendungslogik legt. Expo Router schließt diese Ebene ab, indem 
die Navigationsstruktur direkt aus der Ordnerstruktur abgeleitet wird 
und damit die Nachvollziehbarkeit der Anwendungsarchitektur erhöht.

Auf der Datenschicht ergänzen sich TanStack Query und Zustand 
komplementär: TanStack Query übernimmt das spezialisierte 
Server-State-Management mit automatischem Caching und 
Fehlermanagement, während Zustand den lokalen Client-State 
schlank und ohne Boilerplate verwaltet. Da \textit{petappoint} eine 
datengetriebene Anwendung ist, die primär Serverdaten abruft 
und anzeigt, ist diese Trennung zwischen Server- und
Client-State konsequent umgesetzt.

Gluestack UI und i18next vervollständigen den Stack auf der 
Präsentationsebene: Gluestack UI stellt ein konsistentes 
Token-basiertes Design-System bereit, das ein individuelles 
Erscheinungsbild ohne Bindung an Material Design ermöglicht, 
während i18next die automatische Anpassung der Sprache an die 
Systemsprache des Geräts sicherstellt.

Insgesamt wurde bei der Technologieauswahl darauf geachtet, 
bewährte und gut dokumentierte Lösungen einzusetzen, die sich 
im React Native Ökosystem etabliert haben und komplementär zusammenwirken.
