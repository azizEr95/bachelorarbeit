\section{Technologien in der Datenbankschicht}

Die Datenbankschicht bildet die Grundlage für die Speicherung der Anwendungsdaten, sodass der Verlust jeglicher Daten vermieden wird. Die Daten in einer Datenbank werden anhand des Datenbankmodells geordnet und nach einem vordefinierten Schema strukturiert. In diesen werden einzelne Entitäten und ihre Beziehungen zueinander beschrieben. Diese Sammlung wird üblicherweise von einem Datenbankmanagementsystem (DBMS) verwaltet. Zusammen werden die Daten und das DBMS als Datenbanksystem bezeichnet. Die üblicherweise verwendete Datenbank ist die Relationale Datenbank \cite[S.~4]{halvorsen2017database}. In der relationalen Datenbank werden die Entitäten in Form von Tabellen beschrieben. Jede Entität besitzt einen einzigartigen Primärschlüssel, und dieser Primärschlüssel kann sich in anderen Entitäten wiederfinden. Dieser heißt in der Tabelle Fremsdchlüssel und zeigt auf einer abstrakten Ebene die Beziehung zwischen zwei oder mehreren Tabellen. Dadurch lassen sich Daten ohne Duplikation organisieren \cite{relational_database_explanation}.

\subsubsection{PostgreSQL}

PostgreSQL ist eine relationale Datenbank, die die ACID-Eigenschaften 
(Atomicity, Consistency, Isolation, Durability) gewährleistet und damit 
die Grundlage für sichere und zuverlässige Transaktionen bildet 
\cite{postgresql_about}. Darüber hinaus unterstützt PostgreSQL 
Multi-Table-Transaktionen: Änderungen an mehreren Tabellen werden dabei als eine atomare Einheit ausgeführt. Schlägt eine dieser Änderungen 
eingesetzt, wenn mehrere Entitäten gleichzeitig erstellt, geändert 
fehl, wird die gesamte Transaktion zurückgenommen \cite{postgresql_transactions}.

\section{Technologien in der Backendschicht}
Die Backendschicht dient als zentrale Schnittstelle zwischen Frontend und 
Datenbank. Sie beinhaltet die Geschäftslogik, verarbeitet eingehende 
Anfragen und sorgt für Sicherheit durch Authentifizierung und 
Autorisierung \cite{backend_explanation}.

\subsubsection{Node.js}
Node.js ist eine asynchrone, ereignisgesteuerte JavaScript-Laufzeitumgebung, die für die Entwicklung skalierbarer Netzwerkanwendungen ausgelegt ist. Eingehende Anfragen werden dabei nicht blockierend verarbeitet, sodass mehrere Anfragen gleichzeitig abgearbeitet werden können \cite{nodejs-docs}. 

\subsubsection{Express}
Express ergänzt Node.js um Routing und Middleware und ermöglicht damit 
die strukturierte Verarbeitung von HTTP-Anfragen \cite{express-docs}. 
Gemeinsam bilden Node.js und Express die REST-API, über die das Frontend 
sämtliche Daten abruft und verwaltet.

\subsubsection{TypeScript}
TypeScript ist eine auf JavaScript aufbauende Sprache, die statische Typprüfung für Programme bereitstellt. Im Unterschied zu reinem JavaScript bietet TypeScript Typsicherung und somit eine frühzeitige Fehlererkennung während der Entwicklung \cite{typescript-docs}. Die Typisierung verbessert die Wartbarkeit und reduziert Fehler bei wachsendem Projektumfang.

\subsubsection{Zod}
Zod ist eine Validierungsbibliothek, die deklarative Schemata für 
Datenstrukturen definiert und diese zur Laufzeit überprüft. Sie ist 
eng mit TypeScript verzahnt und stellt sicher, dass alle eingehenden 
API-Anfragen dem erwarteten Format entsprechen \cite{zod-docs}.

\subsubsection{Brevo}
Brevo ist ein Dienst für den automatisierten Versand transaktionaler 
E-Mails über eine API \cite{brevo-docs}. In petappoint wird Brevo 
für den Versand von Buchungsbestätigungen und -erinnerungen eingesetzt.

\section{Technologien in der Frontendschicht}

Die Frontendschicht stellt die Benutzerschnittstelle der Anwendung dar 
und ermöglicht die Interaktion zwischen Nutzer und System. Sie ist 
dafür verantwortlich, Informationen visuell darzustellen und 
Benutzereingaben entgegenzunehmen \cite{frontend_explanation}. 
Im Gegensatz zu den bereits im vorangegangenen Projektmodul 
festgelegten Backend-Technologien wurden die im Folgenden beschriebenen 
Frontend-Technologien im Rahmen dieser Arbeit eigenständig ausgewählt 
und begründet.

latex\subsection{React Native}

React Native ist ein von Meta entwickeltes Open-Source-Framework, das 
die Entwicklung mobiler Anwendungen für iOS und Android auf Basis von 
JavaScript bzw. TypeScript ermöglicht. Im Unterschied zu hybriden 
Ansätzen wie Ionic, die auf WebViews setzen, rendert React Native echte 
native UI-Komponenten, die direkt den nativen Komponenten der jeweiligen 
Plattform entsprechen und damit ein plattformkonformes Nutzererlebnis 
gewährleisten \cite{react-native-docs}.

Ein zentrales Merkmal von React Native ist die sogenannte New 
Architecture, die die interne Kommunikation zwischen JavaScript und 
der nativen Ebene grundlegend überarbeitet hat und ab Version 0.82 
standardmäßig aktiviert ist. Durch das JavaScript Interface (JSI), das neue Rendering-System Fabric sowie lazy geladene TurboModules wird die 
Startzeit, der Speicherverbrauch und die Reaktionsfähigkeit der 
Anwendung spürbar verbessert \cite{react-native-new-arch}.

\subsection{Alternativen zu React Native}

\subsubsection{Flutter}

Flutter ist ein von Google entwickeltes Open-Source-Framework für die 
plattformübergreifende Entwicklung mobiler Anwendungen. Es basiert auf 
der Programmiersprache Dart und verwendet eine eigene Rendering-Engine 
namens Impeller, die UI-Elemente unabhängig von nativen Komponenten 
direkt auf einem Canvas zeichnet. Dies ermöglicht ein konsistentes 
Erscheinungsbild über alle Plattformen hinweg, bedeutet jedoch 
gleichzeitig, dass das plattformspezifische Look \& Feel nicht 
automatisch übernommen wird \cite{flutter-docs}.

\subsubsection{Xamarin / .NET MAUI}

Xamarin ist ein von Microsoft entwickeltes Framework für die 
plattformübergreifende mobile Entwicklung auf Basis von C\# und dem 
.NET-Ökosystem. Mit .NET MAUI wurde Xamarin als Nachfolger 
weiterentwickelt und auf weitere Plattformen ausgedehnt. Beide 
Lösungen ermöglichen die Nutzung nativer UI-Komponenten und sind 
eng in die Microsoft-Entwicklungsumgebung Visual Studio integriert 
\cite{maui-docs}.

\subsection{Auswertung und Begründung}

Wie zu sehen ist, bieten die vorgestellten Frameworks alle ihre Vor- 
und Nachteile. Flutter überzeugt durch ein konsistentes 
plattformübergreifendes Erscheinungsbild, setzt jedoch Dart als 
Programmiersprache voraus, mit der im Rahmen des Studiums und des 
Projektkontexts keine Vorerfahrung bestand. Xamarin bzw. .NET MAUI 
schied aus denselben Gründen aus, da weder C\# noch das 
Microsoft-Ökosystem im Projektverlauf eine Rolle gespielt hatten.

Die Wahl fiel auf React Native, da bereits im Studium sowie im 
vorangegangenen Projektmodul mit React gearbeitet wurde. Da React 
Native auf denselben Konzepten aufbaut, entfiel ein Großteil der 
Einarbeitungszeit. Darüber hinaus wurde für die Desktopansicht von 
petappoint ebenfalls React eingesetzt, sodass Kenntnisse, 
Typdefinitionen und Codestrukturen direkt übertragen werden konnten.

\subsection{Expo}

Expo ist ein Open-Source-Framework, das auf React Native aufbaut und 
die plattformübergreifende Entwicklung für iOS, Android und Web 
vereinfacht. Es abstrahiert die Komplexität nativer Build-Konfigurationen 
und stellt standardisierte Schnittstellen für den Zugriff auf 
gerätespezifische Hardware-Funktionalitäten wie Kamera, Biometrie oder 
sicheren Speicher bereit \cite{expo-docs}. Die offizielle React Native 
Dokumentation empfiehlt Expo ausdrücklich als Einstiegspunkt für neue 
Projekte \cite{react-native-docs}.

Ein zentraler Bestandteil von Expo sind die Expo Application Services 
(EAS), die einen cloudbasierten Build- und Deployment-Prozess 
ermöglichen. Dadurch können native App-Binaries für iOS und Android 
erstellt werden, ohne dass eine lokale Konfiguration von Xcode oder 
Android Studio erforderlich ist \cite{expo-service}.

\subsection{Alternativen zu Expo}

\subsubsection{Bare React Native}

Bare React Native bezeichnet die Verwendung von React Native ohne 
Expo, also ohne das darüberliegende Framework. Dabei hat das 
Entwicklungsteam vollständige Kontrolle über den nativen Code und 
kann plattformspezifische Anpassungen direkt in Xcode bzw. Android 
Studio vornehmen. Dies geht jedoch mit einem deutlich höheren 
Konfigurationsaufwand einher, da sämtliche native Abhängigkeiten 
manuell verwaltet werden müssen \cite{expo-docs}.

\subsection{Auswertung und Begründung}

Da die Wahl bereits auf React Native gefallen war, stellte sich 
lediglich die Frage, ob Expo oder Bare React Native verwendet werden 
soll. Expo wurde gewählt, da es von der offiziellen React Native 
Dokumentation empfohlen wird und den Einstieg in die native 
App-Entwicklung erheblich vereinfacht. Der Managed Workflow von Expo 
eliminiert die Notwendigkeit, Xcode oder Android Studio manuell zu 
konfigurieren, was für ein kleines Entwicklungsteam einen erheblichen 
Zeitvorteil darstellt. Darüber hinaus bietet das Expo SDK eine 
Sammlung geprüfter Bibliotheken für häufig benötigte 
Gerätfunktionalitäten, die ohne zusätzlichen Konfigurationsaufwand 
eingesetzt werden können. Die Expo Application Services ermöglichen 
zudem einen cloudbasierten Build-Prozess, sodass native App-Binaries 
erstellt werden können, ohne auf eine vollständig eingerichtete native 
Entwicklungsumgebung angewiesen zu sein.

\subsection{Expo Router}

Expo Router ist ein dateibasiertes Navigations- und Routing-System, 
das speziell für React Native Anwendungen mit Expo entwickelt wurde. 
Die Navigationsstruktur der Anwendung wird dabei direkt über das 
Dateiverzeichnis abgebildet — jede Datei im \texttt{app}-Verzeichnis 
entspricht automatisch einer Route. Dieses Konzept ist an das Routing 
von Next.js angelehnt und ermöglicht eine intuitive und nachvollziehbare 
Projektstruktur \cite{expo-router}.

Da die Wahl bereits auf Expo gefallen war, ergab sich Expo Router als 
konsequente Folgeentscheidung. Expo Router ist tief in das Expo-Ökosystem 
integriert und wird von Expo als offizielles Navigationssystem empfohlen. 
Die dateibasierte Struktur reduziert den Konfigurationsaufwand für die 
Navigation erheblich und erhöht die Nachvollziehbarkeit der 
Navigationslogik, da die Ordnerstruktur direkt die Struktur der App 
widerspiegelt.

\subsection{TanStack Query}

TanStack Query ist eine Bibliothek für das Management von Server State 
in React- und React-Native-Anwendungen. Sie übernimmt das Abrufen, 
Cachen und Synchronisieren von Serverdaten und reduziert dabei den 
Boilerplate-Code, der bei einer manuellen Implementierung dieser 
Funktionen anfallen würde \cite{tanstack-docs}.

\subsection{Alternativen zu TanStack Query}

\subsubsection{Redux Toolkit Query}

Redux Toolkit Query ist die datenverwaltende Erweiterung von Redux 
Toolkit und bietet ebenfalls Funktionen für das Abrufen und Cachen 
von Serverdaten. Es ist eng in das Redux-Ökosystem integriert und 
eignet sich besonders für Anwendungen, die bereits Redux für das 
State Management einsetzen. Der Einstieg ist jedoch mit einem höheren 
Konfigurationsaufwand und mehr Boilerplate verbunden \cite{redux-docs}.

\subsubsection{Zustand}

Zustand ist eine schlanke State-Management-Bibliothek für React, die 
sich durch eine einfache API und minimalen Boilerplate auszeichnet. 
Sie eignet sich gut für lokalen UI-State, bietet jedoch keine 
spezialisierte Unterstützung für Server State Management oder 
automatisches Caching \cite{zustand-docs}.

\subsection{Auswertung und Begründung}

Die Wahl fiel auf TanStack Query, da die Bibliothek bereits im 
vorangegangenen Projektmodul eingesetzt wurde und damit Vorerfahrung 
im Umgang damit vorhanden war. Darüber hinaus wurde TanStack Query 
vom betreuenden Professor empfohlen. Technisch überzeugt die 
Bibliothek durch ihr spezialisiertes Server State Management und die 
deutliche Reduzierung von Boilerplate-Code gegenüber einer manuellen 
Implementierung. Redux Toolkit Query schied aus, da es einen höheren 
Konfigurationsaufwand mit sich bringt und kein Redux bereits im 
Projekt eingesetzt wurde. Zustand wurde nicht gewählt, da es primär 
für lokalen State ausgelegt ist und keine spezialisierte 
Unterstützung für Server State bietet.

\subsection{NativeWind, Tailwind CSS und Gluestack UI}

Gluestack UI ist eine Komponentenbibliothek für React Native, die 
vorgefertigte, plattformoptimierte UI-Komponenten bereitstellt und 
Design-Themen sowie Dark-Mode-Unterstützung mitbringt 
\cite{gluestack-explanation}. Als Styling-System empfiehlt Gluestack 
UI die Kombination aus NativeWind und Tailwind CSS. NativeWind 
überträgt das Utility-Class-System von Tailwind CSS auf React Native 
und ermöglicht damit ein einheitliches Styling über alle Komponenten 
hinweg \cite{nativewind-explanation}. Tailwind CSS selbst stellt 
dabei die Utility-Klassen bereit, mit denen UI-Elemente direkt im 
Markup gestylt werden können, ohne zwischen separaten Stylesheet-Dateien 
wechseln zu müssen \cite{tailwind-explanation}.

\subsection{Alternativen zu NativeWind, Tailwind CSS und Gluestack UI}

\subsubsection{React Native Paper}

React Native Paper ist eine Komponentenbibliothek für React Native, 
die auf dem Material Design von Google basiert. Sie bietet eine 
Sammlung vorgefertigter Komponenten und unterstützt Theming sowie 
Dark Mode \cite{react-native-paper-docs}.

\subsubsection{StyleSheet}

StyleSheet ist das in React Native integrierte Styling-System. Es 
ermöglicht die Definition von Stilen direkt in JavaScript ohne 
zusätzliche Abhängigkeiten, bietet jedoch keine vorgefertigten 
Komponenten und erfordert bei wachsendem Projektumfang einen höheren 
manuellen Aufwand \cite{react-native-docs}.

\subsection{Auswertung und Begründung}

Die Wahl fiel auf Gluestack UI, da die Bibliothek vom betreuenden 
Professor empfohlen wurde. Da Gluestack UI selbst NativeWind und 
Tailwind CSS als Styling-System empfiehlt, ergab sich die Kombination 
der drei Technologien als logische Einheit. React Native Paper schied 
aus, da es an das Material Design gebunden ist und damit weniger 
Flexibilität in der visuellen Gestaltung bietet. Das integrierte 
StyleSheet-System wurde nicht gewählt, da es keine vorgefertigten 
Komponenten mitbringt und der manuelle Aufwand für ein konsistentes 
UI-Design deutlich höher gewesen wäre.

\subsection{i18next}

i18next ist eine Internationalisierungsbibliothek für JavaScript, die 
in Kombination mit react-i18next für React Native Anwendungen eingesetzt 
wird. Sie ermöglicht die Verwaltung von Übersetzungsdateien und passt 
die Sprache der Oberfläche automatisch an die Systemsprache des Geräts 
an \cite{i18n-explanation}.

Im Vergleich zu react-intl, einer weiteren verbreiteten 
Internationalisierungsbibliothek, zeichnet sich i18next durch eine 
flexible Konfiguration und eine größere Community aus. react-intl ist 
eng an den ECMAScript Internationalization Standard gebunden und eignet 
sich besonders für Web-Anwendungen, bietet jedoch weniger Flexibilität 
bei der Integration in React Native \cite{i18n-explanation}.

Die Wahl fiel auf i18next, da es sich als Standardlösung im React 
Native Ökosystem etabliert hat und die automatische Anpassung an die 
Systemsprache des Geräts ohne zusätzlichen Konfigurationsaufwand 
ermöglicht.

\subsection{TypeScript und Zod}
Da TypeScript bereits im Backend eingesetzt wird, entstand durch die Verwendung im Frontend eine konsistente Typisierung über alle Schichten der Anwendung hinweg. Dasselbe gilt für Zod, da es ebenfalls bereits im Backend eingesetzt wird, können Schemata zwischen Backend und Frontend geteilt werden, was die Konsistenz der Datenvalidierung über die gesamte Anwendung hinweg gewährleistet.

\subsubsection{Expo Secure Store}
Für die Speicherung sicherheitsrelevanter Daten, wie beispielsweise Authentifizierungstokens, nutzt petappoint expo-secure-store. Die Bibliothek legt Daten verschlüsselt als Key-Value-Paare ab und greift dafür direkt auf die systemeigenen Sicherheitsmechanismen der Endgeräte zurück, also die iOS-Keychain oder den Android-KeyStore. Das stellt sicher, dass sensible Informationen lokal auf dem Gerät geschützt bleiben und der Zugriff durch unbefugte Dritte effektiv unterbunden wird \cite{expo-secure-store-docs}.

\section{Auswertung und Begründung der Technologischen Auswahl}

Die gewählte Technologiekombination ergibt sich nicht aus isolierten 
Einzelentscheidungen, sondern aus einem zusammenhängenden Stack, dessen 
Komponenten sich gegenseitig ergänzen und verstärken. Grundlage aller 
Entscheidungen war die Anforderung, eine plattformübergreifende mobile 
Anwendung mit einem kleinen Entwicklungsteam umzusetzen.

React Native bildet das Fundament des Stacks und erfüllt diese 
Anforderung direkt, indem eine einzige Codebasis sowohl für iOS als 
auch für Android genutzt wird. Expo baut darauf auf und reduziert den 
nativen Konfigurationsaufwand erheblich, sodass der Fokus auf die 
eigentliche Anwendungsentwicklung gelegt werden konnte. Expo Router 
ergibt sich als konsequente Folgeentscheidung aus der Expo-Wahl und 
sorgt für eine nachvollziehbare Navigationsstruktur.

Ein zentrales Merkmal des Stacks ist die durchgängige Verwendung von 
TypeScript und Zod über alle Schichten der Anwendung hinweg. Da beide 
Technologien bereits im Backend eingesetzt werden, konnten Typen und 
Schemata zwischen Backend und Frontend geteilt werden. Dies reduziert 
Redundanz und stellt die Konsistenz der Datenstrukturen über die 
gesamte Anwendung sicher.

Darüber hinaus basiert ein Großteil der Technologieentscheidungen auf 
bereits vorhandener Erfahrung. React Native, TanStack Query und 
TypeScript wurden bereits im vorangegangenen Projektmodul eingesetzt, 
sodass die Einarbeitungszeit minimiert werden konnte. Gluestack UI 
und TanStack Query wurden zusätzlich vom betreuenden Professor 
empfohlen, was die Entscheidungssicherheit weiter erhöhte.

Insgesamt bildet der gewählte Stack eine konsistente, 
wartbare und für ein kleines Entwicklungsteam effizient handhabbare 
Grundlage für die Umsetzung von petappoint.
